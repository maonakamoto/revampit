# Translation workflow

**Status (2026-06-03):** DE is the canonical source. Non-DE locales have 186–254 missing keys each. Total: ~1334 missing entries across 6 locales. Page renders use `deepMerge(de, locale)` in `src/i18n/request.ts`, so a missing key falls back to the German string — users see German, never the literal key.

## When to translate

A missing key is **never a crash**. The question is whether the German fallback degrades the experience enough to justify cost:

| Audience | Where they land | Acceptable? |
|---|---|---|
| **Marketplace buyer (FR/IT/EN)** | `/marketplace`, listing detail, checkout | **No** — high commercial impact, translate first |
| **Donor (any locale)** | `/get-involved/donate`, `/projects/*` | **No** — mission-facing |
| **Admin staff** | `/admin/*` | Yes — internal, DE-fluent |
| **About / impact reader** | `/about`, `/transparenz` | **No** — reputation-facing |

The 125 missing **admin** keys per non-DE locale are explicitly acceptable as fallback because admin is DE-only by team convention. Don't pay to translate those.

## Workflow

```bash
# 1. Generate per-locale "missing keys" inventory
npm run i18n:missing
# → writes messages/_missing/<locale>.json (gitignored)

# 2. Send the file to a translator. Format is:
#    { "shop.foo": { "de": "Foo", "translation": null }, ... }
#    Translator fills in every "translation" string.

# 3. Merge translations back:
#    npm run i18n:apply <locale>
#    → safe: never overwrites existing non-empty values; surfaces conflicts
#      (existing differs from candidate) for manual review.
```

## Priority queue (by audience impact)

Generated by namespace, biggest user-impact first:

1. **`marketplace.*`** — every non-DE locale has 5–13 missing keys here. Fast win.
2. **`shop.*`** — 8 missing per non-EN locale, fewer for EN.
3. **`projects.*`** — 25 missing per non-EN locale (the new Monitor-Upcycling content).
4. **`dashboard.*`** — 19 missing per non-EN locale.
5. **`about.*` / `transparenz.*`** — 5–8 missing per locale.
6. **`admin.*`** — skip (DE-only by convention).

## How to receive translator output

Translator returns a filled-in `messages/_missing/<locale>.json`. Merge:

```bash
npm run i18n:apply <locale>
npm run i18n:missing      # verify count dropped
```

Apply-script behavior:
- Skips entries where `translation` is null / empty / whitespace (partial returns are fine).
- Skips keys that already match the candidate (idempotent re-runs).
- Surfaces **conflicts** when an existing translation differs from the candidate — does NOT overwrite; resolve manually then re-run.

## Don't machine-translate UI

Tempting but wrong:
- Google Translate / DeepL output for UI strings produces grammatically-correct but tonally-off translations (formal vs informal, button voice, region quirks like Swiss-German vs DE-DE).
- A mediocre translation is **worse than DE fallback** — users assume the site is in their language and stop noticing weirdness, missing real comprehension issues. DE fallback is honest about the gap.
- The 1334-string load is ~4–8 hours of focused work for a fluent translator with the DE source in front of them — not a script.
