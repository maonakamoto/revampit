Correct, not just convenient

The edge cases most timesheets get wrong.

01

Mid-year workload changes

A 60% → 80% change is effective-dated. Every past month stays correct; nothing is retro-broken.

02

Holidays reduce Soll automatically

A public holiday on a scheduled day drops that day's target to zero — no one enters a holiday row by hand.

03

Paid vs unpaid absences

Paid leave fulfils the target like worked time; unpaid doesn't. Your categories, your rules.

04

Honest historical months

A report for a month before tracking began still shows that month's real hours — never a placeholder zero.

05

As-of reporting

A running month reports as of today. Pre-filled future days never appear as already worked.

06

Bring your own categories

Ships a Swiss default; swap in any set of paid/unpaid categories and holiday calendar.

The engine

Pure functions. Bring your own storage and UI.

One call returns the whole picture. No database, no framework, no globals — just input in, numbers out. Drop it into a Next.js app, a cron job, or a spreadsheet backend.

Install & read the API 
// one call — running balance + month decomposition
import { computeTimeSaldo } from 'saldo-engine'

const r = computeTimeSaldo({
  openingMinutes: 0,
  openingDate:  '2026-01-01',
  periods: [{ validFrom: '2026-01-01', weeklyMinutes: 21*60 }],
  schedule: STANDARD_WEEKLY_SCHEDULE,
  entries: [{ work_date: '2026-07-06', duration_minutes: 420, category: 'work' }],
  holidays: getHolidayDateSet(2026, 2026),
  today:   '2026-07-20',
})

// → { saldoMinutes, monthSollMinutes,
//     monthIstMinutes, monthSaldoMinutes, computedFrom }
